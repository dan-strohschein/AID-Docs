# Benchmark 1: Aria Compiler — Add `spawn` Statement

**Date:** 2026-03-27
**Model:** Claude Sonnet 4.6 (subagents)
**Target codebase:** aria-compiler-go (10.7K LoC Go, novel language compiler)

---

## Test Design

### Objective

Measure whether AID documentation improves an AI agent's ability to implement a feature in a codebase it has never seen before.

### Codebase

The Aria programming language compiler, written in Go. A full compiler pipeline:
- Lexer (tokenization)
- Parser (AST construction)
- Resolver (name/scope resolution)
- Checker (type checking)
- Codegen (code generation)

This is a novel language — the compiler code does not exist in public training data.

### Task

**Add `spawn` statement support to the parser.**

The `spawn` keyword token already exists in the lexer (`lexer.Spawn`, listed as "recognized but unsupported in bootstrap"). The task requires:

1. Add a `SpawnStmt` AST node to `internal/parser/ast.go` following existing conventions (struct with fields, `stmtNode()` interface method, `GetPos()` method)
2. Add a `parseSpawnStmt()` method to `internal/parser/parser.go` that handles `spawn <expression>` and `spawn { block }` syntax
3. Wire the new parser method into the statement parsing switch (alongside `defer`, `return`, etc.)
4. Add the formatter case in the AST pretty-printer

Reference implementation: `DeferStmt` / `parseDeferStmt()` follows the exact same pattern.

### Conditions

| Condition | What the agent receives |
|-----------|------------------------|
| **No AID** | Source code only. No CLAUDE.md, no README.md, no AID files. Must explore the codebase from scratch. |
| **With AID** | Source code + Layer 1 AID files generated by `aid-gen-go` (6 files, 4,324 lines covering all compiler packages). Reads AID first, then source. |

### Methodology

Two Sonnet 4.6 subagents launched in parallel, each in an isolated git worktree (independent copy of the repo). Same task prompt, different documentation. Neither agent can see the other's work.

## Results

### Correctness

| Aspect | No AID | With AID |
|--------|--------|----------|
| AST node struct | `SpawnStmt { Expr Expr; Pos Pos }` | `SpawnStmt { Body Expr; Pos Pos }` |
| Interface methods (`stmtNode`, `GetPos`) | Correct | Correct |
| Parser method (`parseSpawnStmt`) | Correct — mirrors `parseDeferStmt` | Correct — mirrors `parseDeferStmt` |
| Statement switch wiring | Correct — `case lexer.Spawn:` | Correct — `case lexer.Spawn:` |
| Block-or-expression dispatch | Correct — `p.check(lexer.LBrace)` | Correct — `p.check(lexer.LBrace)` |
| AST formatter case | Caught — added `formatStmt` case | Caught — added `formatStmt` case |
| Field naming convention | `Expr` (matches `DeferStmt.Expr` exactly) | `Body` (reasonable but inconsistent with `DeferStmt`) |

**Both agents produced correct, compilable implementations.** The core logic is identical.

### Efficiency

| Metric | No AID | With AID | Difference |
|--------|--------|----------|------------|
| Total tokens consumed | 59,433 | 31,647 | **-47%** |
| Tool calls (file reads, writes) | 24 | 10 | **-58%** |
| Wall clock time | 192 seconds | 33 seconds | **-83%** |

### Behavioral Differences

**No AID agent:** Had to extensively explore the codebase. Made 24 tool calls — reading token.go, ast.go, parser.go, and various other files to understand the patterns before it could write code. Spent most of its time on exploration.

**AID agent:** Read the 6 AID files first (~58KB total vs ~250KB of raw source), immediately understood the compiler architecture, then went directly to the two files that needed modification. Only 10 tool calls total.

---

## Analysis

### What AID improved

1. **Token efficiency: 47% reduction.** The AID files are 23% the size of the raw source but conveyed enough structural information for the agent to understand the codebase architecture without reading every file.

2. **Speed: 6x faster.** The AID agent completed in 33 seconds vs 192 seconds. It skipped the exploration phase entirely because the AID files gave it the architectural overview it needed.

3. **Tool call reduction: 58% fewer.** Fewer file reads means fewer round trips, less context window pressure, and lower cost in a production agentic workflow.

### What AID did NOT improve

1. **Correctness.** Both agents produced correct implementations. The task was well-patterned (mirror an existing feature), and Sonnet's reasoning was strong enough to figure it out from source alone.

2. **Convention adherence.** The no-docs agent actually matched the existing `DeferStmt.Expr` field naming more precisely. The AID agent used `Body` — reasonable but slightly inconsistent. This is a minor point.

### Caveats

- **Single task.** This is one data point. More complex tasks (cross-package changes, semantic analysis, etc.) may show larger correctness differences.
- **Well-patterned task.** Adding `spawn` is a direct mirror of `defer`. Tasks requiring novel logic rather than pattern-following would be a harder test.
- **Same model.** Both agents are Sonnet 4.6. A weaker model might show larger correctness differences with vs without AID.
- **Layer 1 only.** The AID files were mechanical extraction (no workflows, no antipatterns, no semantic constraints). Hand-crafted Layer 2 AID with workflows and antipatterns might improve correctness on harder tasks.

---

## Conclusion

For this task, AID's primary value was **efficiency, not correctness**. The agent with AID docs consumed 47% fewer tokens, made 58% fewer tool calls, and completed 6x faster — while producing equally correct code.

In a production agentic workflow where cost = tokens × price and speed matters, this is a meaningful win. A 47% token reduction across hundreds of tasks in a large codebase compounds into significant cost and time savings.

The true test of AID's correctness value awaits harder tasks that require cross-package understanding, non-obvious constraints, or domain-specific patterns that can't be inferred from code structure alone.
