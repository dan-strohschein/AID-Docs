# Benchmark 2: SyndrDB — BRIN Index Planner Integration

**Date:** 2026-03-27
**Model:** Claude Sonnet 4.6 (subagents)
**Target codebase:** SyndrDB (219K LoC Go, novel database server, 77 packages)

---

## Test Design

### Objective

Measure whether AID documentation improves an AI agent's ability to plan and implement a complex, cross-package feature in a large, novel codebase.

### Codebase

SyndrDB — a custom document database server with:
- Custom query language (SyndrQL) with parser, planner, executor
- Multiple index types (BTree, Hash, BRIN)
- Storage engine with buffer management, WAL, compaction
- GraphQL API layer
- 516 source files, 219K lines of Go

This codebase has never been published or seen in training data.

### Task

**Integrate BRIN index into the query planner's index selection logic.**

A BRIN (Block Range INdex) implementation exists but the planner doesn't properly use it. The task requires:
1. Understanding the BRIN index API (page-range scanning, lossy results)
2. Understanding the planner's index selection logic (cost model, access paths)
3. Understanding how indexes are registered and looked up
4. Fixing the existing integration and extending it for BETWEEN queries

This is a genuinely hard task requiring cross-package understanding of: query planner, cost model, index registry, BRIN index internals, SyndrQL expression system, and execution node architecture.

### Conditions

| Condition | What the agent receives |
|-----------|------------------------|
| **No AID** | Source code only. No CLAUDE.md, no README.md, no AID files. |
| **With AID** | Source code + 69 Layer 1 AID files (57K lines, 1.5MB) generated by `aid-gen-go`. Reads AID first, then source. |

---

## Results

### Both agents discovered the same critical insight

**BRIN infrastructure already partially exists** — `BRINScanNode`, its `Execute` method, and a BRIN block in `tryIndexOptimization` were already present. Both agents independently discovered this and pivoted from "implement from scratch" to "fix and extend what exists."

This is itself a test of codebase understanding — the agent had to find and comprehend existing code before proposing changes.

### Correctness & Completeness

| Issue identified | No AID | With AID |
|-----------------|--------|----------|
| BRIN returns lossy results — needs FilterNode wrapping | Found | Found |
| BETWEEN queries not handled (two-sided range) | Found | Found |
| Cost is hardcoded magic number, not using CostModel | Found | Found |
| Need `BRINScanCost` method on CostModel | Proposed | Proposed |
| Need `ExtractBetweenCondition` helper | Proposed (inline in router) | Proposed (separate function in syndrQL package) |
| Need `isBRINIndexOnField` helper | Not proposed | Proposed |
| BTree vs BRIN cost comparison when both exist | Proposed (inline in BTree block) | Not proposed (kept BTree-first ordering) |
| FilterNode placement | Inside `tryIndexOptimization` | At call site in `createExpressionBasedPlan` |
| Thread `database` param for FilterNode | Identified the dependency | Used existing `database` in outer scope |
| Index registry expansion for BRIN | Proposed `RegisterBRINIndex`/`GetBRINIndex` | Not proposed (relied on existing `IndexInstance`) |

**Both agents identified the same 3 critical issues** (FilterNode wrapping, BETWEEN handling, cost model). They proposed structurally different but equally valid solutions.

### Architectural quality

| Aspect | No AID | With AID |
|--------|--------|----------|
| FilterNode placement | Inside `tryIndexOptimization` — changes function's contract (now returns filtered nodes for BRIN but unfiltered for BTree) | At `createExpressionBasedPlan` call site — cleaner separation (tryIndexOptimization returns raw index nodes, caller decides filtering) |
| BETWEEN detection | Inline helper in `query_router.go` | Separate `ExtractBetweenCondition` in `syndrQL/expression_helpers.go` — better package boundary respect |
| Field matching | Inline conditional | Extracted `isBRINIndexOnField` helper with legacy fallback |
| BTree/BRIN cost comparison | Proposed inner loop comparing both when co-located | Kept existing ordering (BTree first, BRIN fallback) — simpler |

**The AID agent showed better package boundary awareness.** Placing `ExtractBetweenCondition` in the `syndrQL` package (alongside the existing `ExtractRangeCondition`) respects the codebase's separation of concerns. The no-docs agent put everything in the planner package.

**The no-docs agent was more ambitious.** It proposed cost-based selection between BTree and BRIN on the same field, and registry expansion — both valid improvements the AID agent didn't propose.

### Efficiency

| Metric | No AID | With AID | Difference |
|--------|--------|----------|------------|
| Total tokens consumed | 75,311 | 93,475 | **+24%** |
| Tool calls | 37 | 49 | **+32%** |
| Wall clock time | 213 seconds | 208 seconds | **~same** |

**Surprise: the AID agent used MORE tokens and tool calls.**

Why? It read 69 AID files first (~57K lines), adding significant input tokens. For a codebase this large, the AID files themselves are substantial. The no-docs agent was more targeted — it went straight to the relevant source files guided by the task description.

This is an important finding: **Layer 1 AID for a very large codebase can be too much context.** The agent would benefit from selective AID loading (only the 5-6 relevant packages) rather than all 69 files.

## Analysis

### What AID improved

1. **Package boundary awareness.** The AID agent placed `ExtractBetweenCondition` in the correct package (`syndrQL`) rather than inlining it in the planner. The AID files gave it a map of what belongs where.

2. **Cleaner architecture.** FilterNode wrapping at the call site (not inside `tryIndexOptimization`) is a more maintainable design. The AID agent understood the function's contract better.

3. **Helper extraction.** `isBRINIndexOnField` with legacy field-name fallback shows awareness of the index registration patterns. The AID files exposed the `IndexReference` struct's field layout.

### What AID did NOT improve

1. **Token efficiency.** AID actually increased token usage by 24% because the agent read too many AID files. Selective loading would fix this.

2. **Completeness.** The no-docs agent proposed more changes (BTree/BRIN cost comparison, registry expansion). More docs didn't lead to more ideas.

3. **Bug detection.** Both agents found the same critical bugs independently. The model's reasoning about "BRIN is lossy → needs filter" was derived from reading the BRIN implementation, not the AID description.

### Key takeaway

For this complex, cross-package task on a 219K LoC codebase:
- **AID improved architectural quality** (better package boundaries, cleaner abstractions)
- **AID did NOT improve correctness** (both found the same bugs)
- **AID hurt token efficiency** (too much context from 69 AID files)
- **Both agents produced production-quality proposals** — the task was hard enough to be meaningful but both Sonnet agents handled it well

### Recommendations for AID tooling

1. **Selective loading.** Don't load all AID files — let the agent (or a pre-filter) select relevant packages based on the task description. For this task, only ~6 of 69 AID files were relevant.

2. **Layer 2 would help.** Layer 1 AID (mechanical extraction) gives API surface but not semantics. A Layer 2 AID file explaining "BRIN is a lossy index — always filter results" or "tryIndexOptimization returns raw index nodes, not filtered" would have been high-value.

3. **Cross-package workflows.** The biggest gap: no AID workflow describes how the planner selects indexes. This is exactly the kind of tribal knowledge that Layer 2/3 AID captures.
