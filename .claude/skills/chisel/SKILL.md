# Chisel — Semantic Refactoring Tool

Chisel performs precise, codebase-wide refactoring using AID skeletons + Cartograph's semantic graph instead of text-level find-and-replace.

## Prerequisites

- `.aidocs/` directory in the project (generated by `aid-gen-go` or `aid-gen-py`)
- Cartograph binary
- Optional: LSP server for type disambiguation (Go: `gopls`, Python: `pyright`, TS: `typescript-language-server`)

## Commands

### Rename a symbol
```bash
chisel rename <old> <new> -dir .aidocs/ -src src/ -cartograph /path/to/cartograph
```

For method renames, use qualified names:
```bash
chisel rename WALManager.Close WALManager.Shutdown -dir .aidocs/ -src src/ -cartograph /path/to/cartograph
```

### With LSP for disambiguation (recommended for generic method names like Close, Flush, Get)
```bash
chisel rename Head.Close Shutdown -dir .aidocs/ -src . -cartograph /path/to/cartograph -lsp-cmd "gopls serve"
```

### Propagate error returns
```bash
chisel propagate <function> error -dir .aidocs/ -src src/ -cartograph /path/to/cartograph
```

### Move a symbol
```bash
chisel move <symbol> <destination-package> -dir .aidocs/ -src src/ -cartograph /path/to/cartograph
```

## When to Use LSP

Use `-lsp-cmd` when the method name is **generic** (shared by many types):
- `Close`, `Flush`, `Get`, `Set`, `Reset`, `String` — use LSP
- `GetBundleByName`, `CompactStaleHead`, `FlushToDisk` — no LSP needed (distinctive names)

LSP servers by language:
| Language | LSP Command |
|----------|-------------|
| Go | `gopls serve` |
| Python | `pyright-langserver --stdio` |
| TypeScript | `typescript-language-server --stdio` |
| Rust | `rust-analyzer` |
| C/C++ | `clangd` |

If LSP fails to start, chisel falls back to heuristic disambiguation automatically.

## Workflow

1. **Dry-run first** (default) — review the diff output
2. **Check for false positives** — on disambiguation renames, verify the diff doesn't touch unrelated types
3. **Apply** — re-run with `-apply` if the diff looks correct
4. **Verify** — run the project's build command to confirm correctness
5. **Fix gaps** — grep for any remaining references chisel missed

## Flags

| Flag | Description |
|------|-------------|
| `-dir` | Path to .aidocs/ directory |
| `-src` | Path to source tree |
| `-apply` | Actually modify files (default: dry-run) |
| `-cartograph` | Path to cartograph binary |
| `-lsp-cmd` | LSP server command for type verification |
| `-include-comments` | Also rename occurrences in comments |
| `-backup .bak` | Create backup files before modifying |
| `-format json` | Output as JSON instead of unified diff |

## How Chisel Decides: Fast Path vs Graph Path

Chisel automatically chooses the right strategy:

- **Distinctive method names** (3+ camelCase words): Skips cartograph entirely, greps the full source tree. Sub-second performance.
- **Generic method names** (Close, Flush): Uses cartograph's semantic graph to identify the right type's callers, then scopes grep to those files.
- **With LSP**: When heuristics are uncertain about a call site's type, LSP verifies it. This eliminates false positives on generic names.

## Known Limitations

- Without LSP, disambiguation of generic method names relies on variable-name heuristics — may miss call sites where variables don't match the type name
- `propagate` only traces callers that appear in AID skeletons
- AID files must be regenerated after significant code changes
